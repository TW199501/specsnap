# AI Inspector Extension — Creative Vision (v0)

**Status:** Brainstorming / 想像創意階段. Not yet a spec, not yet a plan.  
**Author:** tw199501 + Claude (session 2026-04-19)  
**Scope:** End-to-end vision for an independent, npm-publishable browser extension + core library. No implementation details yet.

---

## 1\. 一句話定位 (One-liner)

> **一個讓「人類用眼睛看 UI」和「AI 動手改 UI」之間的翻譯損耗歸零的檢視器。**
> 
> A zero-loss translator between "human inspects UI visually" and "AI modifies UI precisely".

我們**不是**在做另一個 DevTools。市面上的 inspector（DevTools、Dimensions、VisBug、DOM Extractor for LLMs）各自解了這條路的 \*\*30%\*\*。人類每次都要把「那個按鈕太窄 8px」從眼睛翻成文字，AI 再把文字翻回程式碼，翻譯鏈中間丟失的資訊直接變成 bug。

我們要建的是**中間那層缺失的協定層**：人類把觀察凍結成**結構化 payload**，AI 直接消費，單位明確、座標有參照、命名可信、支援多主體同時對話。

## 2\. 七條北極星原則 (North Star Principles)

**P1 — Viewport 是神聖的。**  
每一份輸出都帶 `viewport: {width, height, devicePixelRatio}` + `scroll: {x, y}`。沒有任何座標可以離開它的參照系單獨出現。這條規則一次消滅 40% 的 AI 誤讀。

**P2 — 批次，不碎片。**  
人類先框（1-N 個），再一次匯出。工具顯示佇列計數徽章，匯出動作是明確的。符合真實的人類思考節奏 ——「我想一次講這三個問題」。

**P3 — 雙語是一等公民 (Bilingual by default)。**  
英文是 AI 的技術詞彙，中文是使用者的閱讀詞彙。每個屬性都帶雙語：`padding: 16px (內邊距)`、`line-height: 1.5 (行高)`。不是 i18n 後掛，是 schema 內建。

**P4 — 一份資料，兩種視圖 (Two views of one truth)。**  
每次匯出同時產出 **Markdown 檔** 和 **標註截圖**。不是二擇一。AI 讀 markdown 做精準比對、人類掃截圖做空間直覺判讀，兩邊資料源同一份、絕不漂移。

**P5 — 遞迴展開，但語意過濾。**  
選中的框會連同**子孫樹**一起 dump，但會濾掉無意義的 wrapper（沒可見文字、沒互動角色、沒非繼承樣式的節點）。Div soup 永遠不進輸出。

**P6 — 輸出即 AI 對話格式 (Conversation-ready)。**  
預設輸出格式就是你複製到 Claude / ChatGPT / Cursor 的格式。Copy-to-clipboard = 立即可貼對話串的 markdown，不需再加工。

**P7 — 只觀察，不改動 (Reversible)。**  
這個工具**絕不**修改頁面。純檢視。改什麼由使用者（和 AI）決定。這條底線讓它在任何頁面都安全可用，包括 production。

## 4\. 使用者故事 (Target UX)

### Happy Path：「Modal 有 3 個問題」

1.  使用者按 `Alt + Shift + I`（可設定）。游標變十字線，螢幕右上浮動面板顯示 **「Inspector mode · 0 frames captured」**。
2.  點擊 modal 的標題區 → **Frame 1** 高亮，tooltip 跳出「Frame 1 captured · 250×40 · padding 16/8/8/16」。
3.  點擊 Port 輸入框 → **Frame 2**。
4.  點擊 Save 按鈕 → **Frame 3**。
5.  按 `E` 匯出 → 浮動面板展開三個按鈕：
    *   📋 **複製為 Markdown**（直接貼到 AI 對話）
    *   🖼️ **複製標註截圖**（帶紅藍框 + px 標籤）
    *   💾 **存檔**：3 個 `.md` + 1 張 `.png` + 1 個 `.json`
6.  使用者貼進 Claude。AI 一次讀完 3 個 frame + viewport 完整脈絡，回覆修改方案。
7.  修改後：**再次檢視同樣三處**，匯出新一份，diff 一目了然 —— 閉環完成。

### Edge Paths

*   **刪除 frame**：hover 已捕獲的框 → 叉號；其他 frame 編號自動重排
*   **重新排序**：拖曳 frame 改順序（影響 MD 檔名 / 輸出順序）
*   **巢狀展開/收合**：預覽面板可控制子樹顯示深度
*   **Viewport 變動警示**：視窗大小在 frame 之間改變 → 匯出頭部加 `⚠ viewport changed: Frame 1-2 @ 1440×900, Frame 3 @ 1280×720`
*   **錯按復原**：`Ctrl + Z` 取消最後一個 frame

### Anti-UX（我們**刻意**不做的）

*   ❌ 編輯 CSS 功能 —— VisBug / Magic CSS 的路線。違反 P7。
*   ❌ 只存 selector 就好 —— 等於退化成 DevTools。
*   ❌ 「AI 自動修」—— 不越界到生成，讓使用者 + AI 保有決策權。

## 8\. MVP vs Roadmap

### MVP（第一版發布目標）

**Must have:**

*   Hotkey 啟動
*   單框 + 多框選取（auto-numbered）
*   Overlay 模式（頁面上直接畫框）+ Side panel Inspector tab（盒模型詳細視圖）
*   Markdown 匯出（含 viewport / frame chain / 盒模型 / 字體 / 色彩）
*   標註截圖匯出
*   JSON 匯出
*   雙語詞庫（前 50 個 CSS 屬性）
*   Copy-to-clipboard
*   Chrome + Edge 支援
*   npm 核心包發布

**Not in MVP:**

*   源碼位置映射（需要 Vite/webpack plugin）
*   組件樹（Vue/React-aware）
*   AI chat 整合按鈕

### V2

*   **Side Panel Colors tab**：頁面 palette 自動抽取 + design token 違規檢測
*   **Side Panel Assets tab**：圖片/字體/icon 清單 + 最佳化提示
*   **Source map 整合**：frame → 準確 `ComponentName.vue:42`
*   **組件階層感知**：不只 DOM，也顯示 Vue / React / Svelte 的組件結構
*   **設計 token 匹配**：若 px 值剛好等於某 CSS variable，自動標註是哪個 token
*   **Diff 模式**：同一 frame 前後兩次匯出可 diff
*   **可設定的語意過濾**（per-project heuristic）
*   Firefox 支援

### V3（aspirational）

*   **Send to Claude Code** 按鈕：一鍵開 conversation，payload 預載
*   **Collaborative mode**：分享 capture URL，同事在自己瀏覽器看到同一份檢視
*   **Timeline mode**：前後多版本對比（「剛剛改了什麼」）
*   **Design system check**：比對 pencil / Figma spec 輸出違規報告

## 10\. 成功指標 — 什麼叫「100 分」

*   **95% 的匯出中，AI 不需要再問 viewport** —— 因為每次都在 payload 裡
*   **前 50 個 CSS 屬性有中文註解** —— 中文使用者零術語斷層
*   **3-frame 典型工作流 \< 10 秒完成匯出**
*   **Dogfood**：antares2 的 CLAUDE.md 加入使用 recipe，第一方使用者就是作者
*   **社群驗證**：發布 30 天內 100+ star；至少 1 個外部 issue / PR
*   **對話實測**：我們自己跟 AI 的對話紀錄中，「找不到元素」這類來回**歸零**

## 附錄 B · 為什麼這個東西會贏

我們贏不是因為有人沒想到，而是因為三件事的**組合**別人沒做：

1.  **多框批次**：市場上的 inspector 都是 conversation（一次一個）；我們是 investigation（先全部框完、再一次交卷）
2.  **雙語輸出**：英文圈開發者不會想到的功能，是中文市場的 unfair advantage
3.  **Viewport 必填**：看似小事，消除 40% 的 AI 誤讀，是**技術細節變產品差異**的範例

這三者任何一個單獨做都不夠，組合在一起才是護城河。

## 附錄 C · Side Panel UX（補充，來自 reference screenshot 2）

> **Scope note：** overlay 模式（在頁面上直接畫框）屬於 **MVP**；side panel 的 Inspector tab 也屬於 MVP（等於 overlay 的 detail view）；**Colors / Assets 兩個 tab 推遲到 V2** —— MVP 階段先專注把 Inspector tab 做到 100 分。

Side panel 分三個 tab：

### Tab 1 · Inspector（核心）

*   盒模型圖示（margin / border / padding 四方向，學 reference 2 的分層配色）
*   元素身份（DOM path + 組件名 + 原始碼位置）
*   分組屬性：Layout / Typography / Background / Interactive States
*   雙語標註

### Tab 2 · Colors

*   頁面上所有用到的色彩 palette（自動去重、按使用頻率排序）
*   標記哪些色彩**不屬於**專案的 CSS token / Tailwind theme（設計系統審計）
*   每個顏色附上使用清單：「`#ff5000` 用在 2 個元素」

### Tab 3 · Assets

*   頁面載入的所有圖片 / icon / font 清單
*   檔案大小、格式、解析度
*   標記未使用 / 可最佳化的項目

這個三分頁結構會讓工具同時**服務視覺 debug**（Inspector）、**設計系統審計**（Colors）、**效能調校**（Assets）三種工作流 —— 一個工具三個使用場景。
